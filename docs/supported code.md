## 1. Введение: цена поддерживаемости

Разработка ПО балансирует между **работоспособностью** (код работает) и **поддерживаемостью** (код не дорого менять).

**Болевая точка:** Существует куча методологий (FSD, Clean Architecture, DDD, SOLID, GRASP, DRY, KISS, YAGNI, GoF, ООП, ФП), но они **противоречат друг другу**. Нельзя взять всё сразу.

---

## 2. Характеристики поддерживаемого кода

> **Поддерживаемый код** = код, в который **не дорого вносить изменения**

Из этого определения вытекают характеристики:

| Характеристика | Суть | Ключевой вопрос |
|:---|:---|:---|
| **Простота** | Читаемость, понятность | Насколько легко понять, что происходит? |
| **Надёжность** | Локализация изменений | Насколько легко найти все места для правок? |
| **Переиспользуемость** | Наличие готовых блоков | Какое количество изменений нужно внести |
| **Абстрактность** | Плотность информации | Сколько строк править при смене требований? |

---

## 3. Простота

> **Простота** — насколько легко понять, что происходит в коде?

### Какую боль решает:
- "Неделю делал MR в одну строчку" — потому что страшно менять непонятный код
- Код покрыт "магией"

### Инструменты для достижения простоты:

| Подход | Как помогает | Какую боль создаёт |
|:---|:---|:---|
| **KISS (Keep It Simple, Stupid)** | Запрещает навороты, заставляет искать простое решение | Может привести к дублированию (страдает переиспользуемость) |
| **DRY (Don't Repeat Yourself)** | Убирает дублирование, код становится короче | Усложняет понимание, если переиспользование надуманное |
| **Отсутствие глубокой вложенности** | Глаз не ломается о 5 уровней if-ов | Иногда требует выноса в отдельные функции |
| **Явные зависимости** | Сразу видно, от чего зависит модуль | Может увеличить объём кода (против DI-контейнеров) |

### Цена простоты:
- Простой код часто менее абстрактный (больше правок при изменениях)
- Простой код может дублироваться (ниже переиспользуемость)

### Где применять:
- Прототипы
- Код, который редко меняется

---

## 4. Надёжность

> **Надёжность** — насколько легко найти все места, где нужно внести изменения?

### Какую боль решает:
- "Сделал MR за день, и неделю правил баги" — потому что не нашёл все места
- Боязнь рефакторинга

### Инструменты для достижения надёжности:

| Подход | Как помогает | Какую боль создаёт |
|:---|:---|:---|
| **SOLID** (особенно Single Responsibility) | Каждая сущность отвечает за одно — легко найти, где править | Классов становится много (страдает простота навигации) |
| **Dependency Injection** | Зависимости явные, можно подменить | Сложность конфигурации, бойлерплейт |
| **GRASP (Information Expert)** | Поведение там, где данные | Может нарушить простоту, если эксперт сложный |
| **Инкапсуляция** | Скрывает детали, меняем в одном месте | Трудно понять, что под капотом |
| **Явные интерфейсы** | Контракт, который нельзя сломать неявно | Нужно проектировать заранее |
| **Статическая типизация** | Компилятор подскажет, где сломалось | Меньше гибкости, больше кода |

### Цена надёжности:
- Больше кода (интерфейсы, классы, DI)
- Сложнее навигация (много мелких файлов)

### Где применять:
- Критическая бизнес-логика
- Ядро системы
- Код, который часто меняется разными людьми

## 5. Переиспользуемость

> **Переиспользуемость** — можно ли использовать существующий код или писать с нуля?

### Какую боль решает:
- "Неделю добавлял новую страницу с нуля" — потому что нет готовых компонентов
- Копипаста и размножение сущностей
- Невозможность быстро прототипировать

### Инструменты для достижения переиспользуемости:

| Подход | Как помогает | Какую боль создаёт |
|:---|:---|:---|
| **DRY** | Запрещает копипасту, заставляет выносить общее | Может создать жёсткие связи там, где не надо |
| **UIKit / Component Libraries** | Готовые кирпичики для интерфейса | Трудно подстроить под специфику |
| **Composition over Inheritance** | Гибкое комбинирование поведения | Сложнее, чем просто унаследовать |
| **GRASP (Low Coupling)** | Слабо связанные модули можно переиспользовать где угодно | Требует правильного выделения границ |
| **GRASP (High Cohesion)** | Внутри модуля всё связано — легко взять целиком | Модуль может быть слишком большим |
| **Паттерны GoF** | Проверенные решения типовых задач | Много кода ради паттерна |
| **Библиотеки** | Чужой код, который решает твою задачу | Зависимость от чужого API и багов |

### Цена переиспользуемости:
- Код становится более абстрактным (меньше конкретики)
- Сложнее понять, как он работает внутри
- Жёсткие связи, если переиспользование надуманное

### Где применять:
- UI-компоненты
- Утилиты и хелперы
- Бизнес-логика, которая повторяется в разных местах

## 6. Высокий уровень абстракции

> **Абстрактность** — сколько строк нужно поправить, чтобы изменить информацию о системе?

**Суть:** Чем выше уровень абстракции, тем больше информации несёт строчка кода.

**Пример из презентации:** 1000 строк на C несут меньше информации о бизнес-логике, чем 1000 строк JavaScript с фреймворками.

### Какую боль решает:
- "Новая роль = 300 файлов" — потому что изменение требует правки в 300 местах
- Низкая скорость разработки
- Много бойлерплейта

### Инструменты для повышения абстракции:

#### 6.1. Использование высокоуровневых решений

| Инструмент | Как повышает абстракцию | Цена |
|:---|:---|:---|
| **Next.js** | Спрятал роутинг, SSR, сборку | Ограничения кастомных решений |
| **React-query** | Спрятал кэширование, ретраи, статусы запросов | Мало контроля над деталями |
| **Material UI / shadcn/ui** | Готовые компоненты с поведением | Трудно кастомизировать |
| **NextAuth** | Готовая авторизация через Google/GitHub | Сложно встроить свою логику |

> **Правило:** Используй, пока не встретил ограничений по работоспособности

#### 6.2. Создание своей инфраструктуры

**Проблема готовых библиотек:** Они предлагают "удобный" интерфейс для всех, но он не идеален ни для одной конкретной задачи.

**Решение:** Зная свои требования, напиши свою абстракцию.

| Пример своей инфраструктуры | Какую боль решает |
|:---|:---|
| **Билдер форм** | Не писать однотипную валидацию и стейт для каждой формы |
| **Строго типизированные query-параметры** | Не парсить строки вручную, типы на уровне кода |
| **Генератор админки из Swagger** | Админка автоматически подстраивается под API |
| **Рендерилка SVG-стрелок** | Не считать координаты вручную |

> **Важно:** Создание инфраструктуры — это сложно, но это то, как сеньор может оптимизировать работу джунов. Инфраструктура позволяет писать **меньше** кода, который становится более **простым** в использовании.

#### 6.3. Создание DSL (Domain Specific Language)

**DSL** — язык для высокоуровневого описания решения конкретной задачи.

| Тип | Примеры | Как помогает |
|:---|:---|:---|
| **Внутренний** (на синтаксисе языка) | JSX, React Router, Prisma, XState, Jest | Описываем UI/маршруты/состояния декларативно |
| **Внешний** (другой синтаксис) | SQL, HTML, CSS | Специализированные языки для своих задач |

**Особенности DSL:**
- Часто **декларативны** (описываем "что", а не "как")
- Нет последовательности выполнения (всё сразу)
- Можно использовать даже для одноразовых модулей

### Цена абстракции:
- Сложно дебажить (что под капотом?)
- Порог входа для новых разработчиков
- Можно переабстрагировать и усложнить код

### Где применять:
- Повторяющиеся паттерны в коде
- Сложные, но чётко определённые области (формы, состояния, запросы)
- Когда хочется "идеально" и есть дохрена времени и денег

## 7. Инструменты и их влияние на характеристики

В презентации перечислены методологии. Вот как они влияют на 4 характеристики:

| Инструмент | Улучшает | Ухудшает | Комментарий |
|:---|:---|:---|:---|
| **FSD** | Надёжность, Абстракцию | Простоту | Чёткая структура, но много слоёв |
| **Clean Architecture** | Надёжность, Переиспользуемость | Простоту | Много слоёв и интерфейсов |
| **DDD** | Абстракцию, Надёжность | Простоту | Сложный язык, требует погружения |
| **SOLID** | Надёжность, Переиспользуемость | Простоту | Много маленьких классов |
| **GRASP** | Надёжность, Переиспользуемость | Простоту | Требует анализа ответственности |
| **DRY** | Переиспользуемость | Простоту | Может создать жёсткие связи |
| **KISS** | Простоту | Переиспользуемость | Может привести к дублированию |
| **YAGNI** | Простоту | Абстракцию | Не делаем лишнего, но потом переписываем |
| **GoF** | Переиспользуемость | Простоту | Паттерны усложняют код |
| **ООП** | Надёжность, Переиспользуемость | Простоту | Инкапсуляция, но много бойлерплейта |
| **ФП** | Надёжность, Абстракцию | Простоту | Чистые функции, но сложные монады |

## 8. Главное правило: баланс и противоречия

Четыре характеристики постоянно противоречат друг другу:

`ПРОСТОТА ↔ НАДЁЖНОСТЬ ↔ ПЕРЕИСПОЛЬЗУЕМОСТЬ ↔ АБСТРАКТНОСТЬ`

**Как это выглядит в жизни:**

| Противоречие | Пример |
|:---|:---|
| Простота ↔ Надёжность | Простой монолит искать легко, но менять страшно. Разделил на микросервисы — надёжно, но сложно понять систему целиком |
| Простота ↔ Переиспользуемость | Написал конкретный код под задачу — просто. Сделал универсальный компонент — сложнее, но можно переиспользовать |
| Простота ↔ Абстрактность | Конкретный код понятен, но при изменении правишь 100 мест. Абстрактный код сложен, но правишь 1 место |
| Надёжность ↔ Переиспользуемость | Связал всё жёстко — надёжно, но не переиспользовать. Сделал слабые связи — переиспользуется, но сложнее гарантировать надёжность |

### Стратегии выбора:

1. **Ищи баланс** — старайся, чтобы всего было понемножку
2. **Действуй от ситуации**:

| Ситуация | Приоритет |
|:---|:---|
| Критическая бизнес-логика | Надёжность + Переиспользуемость |
| Экспериментальный прототип | Простота (можно выбросить) |
| Библиотека для других | Переиспользуемость + Абстрактность |
| Легаси-проект | Надёжность (чтобы не сломать) |
| Стартап (time-to-market) | Простота + Скорость (жертвуем абстракцией) |
